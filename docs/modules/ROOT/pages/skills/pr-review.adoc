= PR Review

NOTE: This documentation was generated with the assistance of AI. Please report any inaccuracies.

Review a specific, already-open GitHub pull request and leave the findings as review comments on it — this
skill never edits code itself. It is grounded in this repository's actual architecture and conventions,
learned fresh via the `explore` skill, plus the originating issue's intent when the PR links one.

== Purpose

`pr-review` fetches a pull request's metadata, diff, and existing comments by id, resolves any linked
GitHub issue (explicitly passed, or detected from a `Closes #N`-style reference in the PR title/body), and
runs `explore` (scoped to that issue when there is one) so the review is grounded in how the codebase is
actually structured rather than the diff in isolation. Where the external `code-review` skill happens to be
available in the current session, `pr-review` prefers delegating correctness/reuse/simplification/efficiency
analysis to it — checking out the PR locally and invoking it with `--comment` — rather than reimplementing
that logic, and focuses its own review on what `code-review` has no way to know: whether the change matches
the linked issue's intent, and whether it follows this codebase's own conventions. If `code-review` isn't
available, `pr-review` covers the full scope itself. Findings are always drafted first and only posted as
inline review comments after the user's explicit confirmation.

|None — must be provided or supplied interactively.

|`issue-id`

== Outputs

* A drafted set of review findings in the conversation, grouped by file, each with its line, a one-sentence
  summary, and a concrete suggested fix, plus a short overall summary.
* If the user confirms posting: a single pull request review (one review event, `event: "COMMENT"`, never
  `REQUEST_CHANGES`/`APPROVE`) bundling every confirmed finding as inline comments, submitted via
  `gh api .../pulls/<pr-id>/reviews`, with its URL reported.
* If `code-review` was available and the user confirmed delegating to it, its own findings are posted
  directly to the PR as part of Step 5 (separately from, and before, this skill's own review in Step 8).
* If the user declines to post: no PR state is changed — the drafted findings are the deliverable.
* This skill never edits code and never approves or requests changes on a PR.

== Dependencies

=== Invokes

* xref:skills/explore.adoc[] — Step 4 runs `explore` (scoped to the linked issue when one was found in Step
  3) to learn the codebase's architecture and conventions before judging the diff against them.
* `code-review` — an external skill, **not part of this catalog**. Step 5 checks whether it happens to be
  available in the current session and, if so, checks out the PR locally and delegates
  correctness/reuse/simplification/efficiency findings to it via its `--comment` flag (which posts those
  findings directly to the PR). If it isn't available, `pr-review` covers that scope itself in Step 6.

=== Invoked by

* xref:skills/issue.adoc[] — Step 7.6 runs `pr-review`, passing both the new PR's id and the issue id
  explicitly, since `issue` already knows the relationship between them with certainty.

=== Related agents

None directly — `pr-review` does not spawn any agent itself (the `explore` skill it invokes in Step 4 is
the one that uses the built-in `Explore` agent).
