= Explore

NOTE: This documentation was generated with the assistance of AI. Please report any inaccuracies.

Produce a grounded understanding of this repository's codebase and, when available, of a specific GitHub
issue — without writing a plan or making code changes. This is read-only research: its output is a report,
not a diff.

== Purpose

`explore` gets an agent (or a human) oriented before any decision or code change is made. Given a GitHub
issue id, it narrows the investigation to only the areas that issue implicates; given none, it does a
general architectural tour of the repository. Either way it never edits a file, and it never proposes an
implementation — that is `plan`'s job, run as a separate, explicit step.

== Invocation

[source]
----

----

== Inputs

[cols="1,1,3,2", options="header"]
|===
|Argument |Required |Description |Default

|`issue-id`
|No
|A GitHub issue number to ground the exploration in. When given, the investigation is scoped to the files,
classes, and behavior the issue describes.
|None — if omitted, the skill asks whether to explore a specific issue anyway; declining falls back to a
general codebase tour.
|===

== Outputs

* A plain-text report in the conversation — never a written file.
* If an issue was explored: the task restated in plain language, the specific files/classes/methods
  relevant to it, how the current code behaves in that area, and anything ambiguous or missing that would
  block implementation.
* If codebase-only: a summary of the architecture areas covered and anything notable found.
* No code, configuration, or documentation is ever modified.
