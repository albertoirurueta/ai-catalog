= Issue

NOTE: This documentation was generated with the assistance of AI. Please report any inaccuracies.

Take a tracked ticket — a GitHub issue or a Jira ticket — from "just filed" to "pull request open for review,
with review comments already left" in one pass. This skill orchestrates `explore`, `plan`, `code`,
`pr-description`, and `pr-review` — it does not duplicate their logic, it sequences them around branch creation,
ticket triage, and PR creation.

== Purpose

`issue` is the top-level, end-to-end entry point for working a tracked ticket, auto-detecting whether it's a
GitHub issue or a Jira key the same way `explore` does. Unlike `explore` and `plan`, it requires a ticket id and
stops immediately if none is given. It fetches the ticket, classifies it as a feature or a hotfix, creates the
matching branch, runs `explore` and `plan` to produce a reviewable `implementation_plan.md`, then — depending on
the user's choice — either stops for manual review or hands implementation off to `code` inside a sub-agent,
before pushing the branch, opening the pull request (as a draft, using whichever tooling matches the detected
repository host), drafting its description with `pr-description`, and leaving an initial review on it with
`pr-review`. If `code`'s security gate (backed by `check-security`) ever fires during implementation — even if
`code` goes on to resolve it and finish cleanly — this skill treats that as a hard stop distinct from an ordinary
blocker: it warns the user with the specifics and never pushes the branch or opens a PR for that run.

== Outputs

* A new local branch, `feature/<ticket-id>` or `hotfix/<ticket-id>` depending on classification, created off
  the branch `/issue` was invoked from.
* `implementation_plan.md` at the repository root (produced by `plan`), ready for manual review, or already
  executed by `code` if the user chose automatic implementation.
* If implementation proceeded automatically and completed successfully, with `code`'s security gate never
  firing: the branch pushed to `origin`, a pull request opened **as a draft** back to the base branch (title
  `"<ticket-title> (#<ticket-id>)"` or `"(<ticket-id>)"` for Jira, body linking the ticket via the convention
  matching the detected host), its description filled in by `pr-description`, and an initial review left on it
  by `pr-review`.
* If the user chose manual review instead, `code` reports a blocker, or `code`'s security gate ever fired during
  the run, the skill stops short of pushing or opening a PR and reports exactly where it stopped — for the
  security-gate case, with an explicit warning naming what was flagged and reminding the user to verify the
  resolution themselves before pushing anything.
* A final summary: ticket id/title, classification and why, the branch and its base, and one of: manual review
  pending, the security gate fired (with the sub-agent's implementation summary and what was flagged), or the
  full implementation/PR/review outcome — always noting the PR is a draft still needing the user's own review.

== Dependencies

=== Invokes

* xref:skills/explore.adoc[] — Step 5 runs `explore` scoped to the ticket id as the first step of the
  explore → plan → code → PR pipeline.
* xref:skills/plan.adoc[] — Step 5 runs `plan` scoped to the ticket id right after `explore`, producing
  `implementation_plan.md`.
* xref:skills/code.adoc[] — Step 6, when the user chooses automatic implementation over manual review, runs
  `code` via the xref:agents/isolated-skill-executor.adoc[`isolated-skill-executor`] agent so the whole
  implementation run starts from a clean context rather than carrying the exploration/planning transcript into it.
* xref:skills/pr-description.adoc[] — Step 7 invokes it to draft the new pull request's description from
  the actual diff, then re-checks its output still references the ticket.
* xref:skills/pr-review.adoc[] — Step 7 invokes it, passing both the new PR's id and the ticket id
  explicitly (rather than relying on its own auto-detection), to leave an initial review on the PR.

=== Invoked by

None — `issue` is a top-level entry point; no other skill in this catalog invokes it, though
xref:skills/create-github-issue.adoc[] and xref:skills/create-jira-ticket.adoc[] each file a ticket meant to be
picked up by a subsequent, separate `/issue` invocation.

=== Related agents

* xref:agents/isolated-skill-executor.adoc[`isolated-skill-executor`] — wraps the entire `code` run in Step 6,
  so implementation happens in an isolated context and this skill can keep running afterward to open the pull
  request in Step 7.
