= Issue

NOTE: This documentation was generated with the assistance of AI. Please report any inaccuracies.

Take a GitHub issue from "just filed" to "pull request open for review, with review comments already
left" in one pass. This skill orchestrates `explore`, `plan`, `code`, `pr-description`, and `pr-review` — it
does not duplicate their logic, it sequences them around branch creation, issue triage, and PR creation.

== Purpose

`issue` is the top-level, end-to-end entry point for working a tracked GitHub issue. Unlike `explore` and
`plan`, it requires an issue id and stops immediately if none is given. It fetches the issue, classifies it
as a feature or a hotfix, creates the matching branch, runs `explore` and `plan` to produce a reviewable
`implementation_plan.md`, then — depending on the user's choice — either stops for manual review or hands
implementation off to `code` inside a sub-agent, before pushing the branch, opening the pull request,
drafting its description with `pr-description`, and leaving an initial review on it with `pr-review`.

== Outputs

* A new local branch, `feature/<issue-id>` or `hotfix/<issue-id>` depending on classification, created off
  the branch `/issue` was invoked from.
* `implementation_plan.md` at the repository root (produced by `plan`), ready for manual review, or already
  executed by `code` if the user chose automatic implementation.
* If implementation proceeded automatically and completed successfully: the branch pushed to `origin`, a
  pull request opened back to the base branch (title `"<issue-title> (#<issue-id>)"`, body linking
  `Closes #<issue-id>`), its description filled in by `pr-description`, and an initial review left on it by
  `pr-review`.
* If the user chose manual review instead, or if `code` reports a blocker, the skill stops short of pushing
  or opening a PR and reports exactly where it stopped.
* A final summary: issue id/title, classification and why, the branch and its base, and either the manual
  review pending state or the full implementation/PR/review outcome.

== Dependencies

=== Invokes

* xref:skills/explore.adoc[] — Step 5.1 runs `explore` scoped to the issue id as the first step of the
  explore → plan → code → PR pipeline.
* xref:skills/plan.adoc[] — Step 5.2 runs `plan` scoped to the issue id right after `explore`, producing
  `implementation_plan.md`.
* xref:skills/code.adoc[] — Step 6, when the user chooses automatic implementation over manual review, runs
  `code` via a `general-purpose` sub-agent (see xref:agents/overview.adoc[]) so the whole implementation run
  starts from a clean context rather than carrying the exploration/planning transcript into it.
* xref:skills/pr-description.adoc[] — Step 7.4 invokes it to draft the new pull request's description from
  the actual diff, and Step 7.5 checks its output still references the issue.
* xref:skills/pr-review.adoc[] — Step 7.6 invokes it, passing both the new PR's id and the issue id
  explicitly (rather than relying on its own auto-detection), to leave an initial review on the PR.

=== Invoked by

None — `issue` is a top-level entry point; no other skill in this catalog invokes it.

=== Related agents

* `general-purpose` (see xref:agents/overview.adoc[]) — one such sub-agent wraps the entire `code` run in
  Step 6, so implementation happens in an isolated context and this skill can keep running afterward to
  open the pull request in Step 7.
