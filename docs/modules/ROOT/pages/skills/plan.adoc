= Plan

NOTE: This documentation was generated with the assistance of AI. Please report any inaccuracies.

Produce a concrete, step-by-step implementation plan for a task — grounded in the real state of the
codebase — and write it to `implementation_plan.md` at the repository root. This skill does not modify
source code; its only output artifact is the plan file itself.

== Purpose

`plan` turns a GitHub issue or a free-form task description into a reviewable, checkbox-driven list of
implementation steps before any code changes begin. It grounds itself in the actual codebase (via
`explore`), resolves only genuinely ambiguous decisions with the user, and determines which
`<key>-code-one-task` skill (e.g. `java-code-one-task`, `dotnet-code-one-task`) each task belongs to — once for
the whole plan if uniform, or per top-level task if the plan spans more than one language/framework — so the
`code` skill that later executes it knows exactly which skill to dispatch each task to without re-inferring it.
Everything is written to `implementation_plan.md` — the artifact `code` later executes task by task.

== Outputs

* `implementation_plan.md` at the repository root, overwriting any existing file at that path (with a
  check-with-the-user safeguard if the existing file looks like in-progress work).
* The plan is structured as: a task summary (including a plan-wide **Language/framework** line if every task
  shares one key), a summary of the current relevant code state (concrete file paths and class/method names),
  and a numbered list of implementation steps — each task and sub-task carrying its own empty markdown checkbox
  (`[ ]`) for `code` to check off later, and tagged with its own language/framework key inline if the plan spans
  more than one.
* The distinct set of `<key>-code-one-task` keys involved, resolved against which such skills are actually
  installed in this repository (`find .claude/skills -maxdepth 1 -type d -name "*-code-one-task"`) — a task
  whose language/framework has no matching skill installed is still recorded accurately, just without a
  dispatchable tag.
* Any decisions made on the user's behalf (per Step 3) are stated in the plan itself, with a one-line
  rationale, rather than asked upfront.
* No source code, configuration, or documentation outside `implementation_plan.md` is ever modified.
* A short summary in the conversation once the plan is written, asking the user to review it before
  implementation starts.

== Dependencies

=== Invokes

* xref:skills/explore.adoc[] — Step 2 runs `explore` (scoped to the issue id, if any) to ground the plan in
  the real codebase, unless equivalent exploration already happened earlier in the conversation.

=== Invoked by

* xref:skills/issue.adoc[] — Step 5 runs `plan` (scoped to the ticket id) right after `explore`, as part of
  its explore → plan → code → PR pipeline. After a reviewable `implementation_plan.md` exists, `issue`
  offers to hand the resulting plan off to `code` for automatic implementation.

=== Related agents

`plan` itself does not spawn any Agent — it calls the `explore` skill directly and otherwise only reads the
codebase and writes the plan file. However, the `implementation_plan.md` it writes explicitly instructs
whoever executes that plan (normally the `code` skill, via the `<key>-code-one-task` skill it dispatches to) to
delegate each task's test run to the xref:agents/gate-runner.adoc[`gate-runner`] agent, if installed, rather
than running tests inline, so test output doesn't consume the executing agent's main context window. This is an
instruction `plan` embeds in its output file for a later execution step to follow — not an `Agent()` call `plan`
makes itself.
