= .NET Code One Task

NOTE: This documentation was generated with the assistance of AI. Please report any inaccuracies.

Carry out a single task from a .NET `implementation_plan.md`-style task list end to end: implement it, back it
with test-framework tests (xUnit/NUnit/MSTest, whichever the project uses), keep license headers and XML doc
comments current, verify nothing regressed, and report the outcome. This is the .NET/C# counterpart to
`java-code-one-task` — same shape, but built on `dotnet-code-quality`, `dotnet-test`, `dotnet-coverage`, and
`dotnet-docfx` instead of the Maven/Java tool chain.

== Purpose

`dotnet-code-one-task` is the .NET-specific worker behind a `code`-style skill's per-task dispatch. Given one
task's own text (what to build, the named file(s)/type(s)/method(s), the described behavior, and any sub-tasks),
it re-checks the current code state, implements exactly what's specified, writes/updates tests in whichever
framework this project already uses, adds license headers via `check-license`, updates XML doc comments via
`dotnet-docfx`, runs the scoped tests and checks coverage via `dotnet-test`/`dotnet-coverage`, runs the full
suite, and checks for new static-analysis issues via `dotnet-code-quality` against a baseline it captures
itself — all through sub-agents so the noisy report output from each stays out of its own context. It hands back
only a short summary; all plan-file bookkeeping (checking boxes, tracking session tasks) is the caller's job, not
this skill's.

== Outputs

* Source and test files created/modified per the task, following this repository's conventions from `CLAUDE.md`
  (target framework/language version, `var`/nullable reference types only where already used, full XML doc
  comments, `ArgumentException`/`ArgumentNullException` for invalid arguments, no new NuGet dependencies).
* Tests added/updated in this project's existing test framework, achieving at least 80% line coverage for the
  changed/new classes, verified against the actual `dotnet-coverage` report rather than estimated.
* License headers added to every file the task touched, via `check-license`.
* XML doc comments added/updated for every file the task touched, via `dotnet-docfx`, verified by a DocFX build
  where `docfx` is installed.
* Confirmation that the scoped tests and the full suite both pass, and that no new `dotnet-code-quality` issue
  was introduced versus a pre-change, per-type baseline this skill captures itself.
* A short summary handed back to the caller: file(s) touched, tests added/updated, coverage achieved, and the
  code-quality outcome — including any new issue left in place as unavoidable and why, whether license-header
  generation was skipped by the user, or whether the run stopped on a blocker.
* `implementation_plan.md` is never read or written by this skill — checking off boxes is the caller's job.

== Dependencies

=== Invokes

Each of the following is invoked via its own `general-purpose` sub-agent (see xref:agents/overview.adoc[]) rather
than directly, so verbose report output stays out of this skill's own context window and only the distilled
outcome comes back:

* xref:skills/dotnet-code-quality.adoc[] — Step 1, to capture a per-type pre-change quality baseline, and Step 9,
  to diff the post-change result against it and surface only newly-introduced issues.
* xref:skills/check-license.adoc[] — Step 4, to add/verify license headers on the file(s) this task touched.
* xref:skills/dotnet-docfx.adoc[] — Step 5, to update XML doc comments for the type(s) this task touched and
  confirm the DocFX build passes (or was skipped because `docfx` isn't installed).
* xref:skills/dotnet-test.adoc[] — Step 6, to run the scoped tests for the affected type(s) (falling back to
  `dotnet test --filter "FullyQualifiedName~<TestClass>"` directly if unavailable).
* xref:skills/dotnet-coverage.adoc[] — Step 7, to check line/branch coverage for the changed/new class(es) is at
  least 80% (falling back to `dotnet test --collect:"XPlat Code Coverage"` directly if unavailable).
* Plain, non-skill sub-agent call: a full `dotnet test` run (Step 8).

=== Invoked by

* xref:skills/code.adoc[] — Step 4 invokes `dotnet-code-one-task` once per top-level task tagged (or defaulted to)
  the `dotnet` language/framework key, passing that task's full text, then checks off its box in
  `implementation_plan.md` on a clean report.

=== Related agents

* `general-purpose` (see xref:agents/overview.adoc[]) — spawned repeatedly, once per delegated check (baseline
  capture, license, doc comments, scoped tests, coverage, full suite, quality diff), for the same reason
  `java-code-one-task` and `code` itself spawn one per delegated check: keeping verbose report/build output out of
  this skill's own context.
