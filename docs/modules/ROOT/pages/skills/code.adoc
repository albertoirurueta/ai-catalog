= Code

NOTE: This documentation was generated with the assistance of AI. Please report any inaccuracies.

Execute `implementation_plan.md` end to end: dispatch each listed task to the language/framework-specific
`<key>-code-one-task` skill the plan declares, capture and re-check project-wide quality and security baselines,
check off the plan's checkboxes as tasks complete, update the project's documentation, and archive the plan. This
skill orchestrates the actual implementation work; it is the execution counterpart to the `plan` skill.

== Purpose

`code` is the autonomous orchestration engine of the catalog. Given a plan produced by `plan` (or by `issue`,
which runs `plan` internally), it determines which language/framework key(s) are in play, captures a
project-wide code-quality baseline (via `java-code-quality`, `dotnet-code-quality`, or whichever
`<key>-code-quality` skill matches) and a project-wide security baseline (via `check-security`), then works
through every task in plan order — dispatching each to its matching `<key>-code-one-task` skill (e.g.
`java-code-one-task`, `dotnet-code-one-task`), which itself implements, tests, license-headers, and documents the
task and hands back a short summary — checking off the plan's checkboxes as it goes so the run is resumable if
interrupted. A task with no matching `<key>-code-one-task` skill installed (or no determinable language/
framework) is implemented directly by `code` itself instead. On successful completion it runs a full local
verification build, re-checks the quality and security baselines (the security gate is a hard stop on any
increase, unlike the quality gate), updates the project's Antora documentation via `update-docs`, archives the
plan to `.archive/`, and asks the user for a final review; it never commits, pushes, or opens a PR itself.

== Outputs

* Source and test files created/modified per each task in `implementation_plan.md`, via the dispatched
  `<key>-code-one-task` skill (or implemented directly when no key/matching skill applies), following this
  repository's own conventions.
* Tests added/updated per task achieving at least 80% line coverage for the changed/new classes, verified against
  the real coverage report rather than estimated — the responsibility of the dispatched one-task skill.
* License headers and language-appropriate documentation (Javadoc, XML doc comments, etc.) kept current per task,
  also delegated to the one-task skill.
* `implementation_plan.md` with each completed task's (and sub-task's) checkbox checked (`[ ]` → `[x]`) and a
  short note naming the files touched, tests added, coverage achieved, and code-quality result.
* A project-wide quality baseline (Step 3) compared against the final state (Step 7) per language/framework key
  involved — if it regressed, the user is asked to add follow-up tasks, file GitHub issues, or accept it as-is
  before archiving.
* A project-wide security baseline (Step 3, via `check-security`) compared against the final state (Step 7) — any
  increase is a hard stop: the user must resolve it (remove/rotate the secret, or audit a false positive
  themselves) before the run can proceed to archiving.
* A full local verification build confirmed healthy before archiving.
* The project's Antora documentation brought up to date via `update-docs` (Step 8), covering everything this run
  changed — or a plain note that there's no Antora module to update.
* On success: `implementation_plan.md` moved to `.archive/implementation_plan_<issue-id-or-timestamp>.md`, a
  summary of tasks completed/files touched/coverage/quality/security outcome/documentation updates, and a request
  for the user to review the code changes — no commit, push, or PR is ever made by this skill.
* On an unresolved blocker: the plan stays unchecked and at the repository root (not archived) so a later run can
  resume it.

== Dependencies

=== Invokes

* xref:skills/java-code-one-task.adoc[] — Step 4, once per top-level task tagged (or defaulted to) the `java`
  key, passing that task's full text and any relevant code-state context, directly via `Skill` (not wrapped in a
  sub-agent) — this skill re-checks the code state, implements, tests, and documents that one task and hands back
  a short summary.
* xref:skills/dotnet-code-one-task.adoc[] — the same role as `java-code-one-task`, for tasks tagged `dotnet`.
* xref:skills/update-docs.adoc[] — Step 8, once per run (not per task), invoked directly via `Skill` once the
  build, quality, and security gates all pass, so the project's documentation reflects everything the whole plan
  changed.

Each of the following is invoked via its own `general-purpose` sub-agent (see xref:agents/overview.adoc[])
rather than directly, so verbose report output stays out of `code`'s own context window and only the distilled
outcome comes back:

* xref:skills/java-code-quality.adoc[] / xref:skills/dotnet-code-quality.adoc[] — Step 3 (project-wide baseline,
  once per language/framework key involved) and Step 7 (final project-wide comparison against that baseline).
* xref:skills/check-security.adoc[] — Step 3 (project-wide baseline) and Step 7 (final comparison) — any increase
  is a hard stop, unlike the quality gate's "accept as-is" option.
* Plain, non-skill sub-agent calls: a final full-build verification (Step 7) via `mvn clean jacoco:prepare-agent
  install jacoco:report javadoc:jar source:jar -P '!build-extras'` (or the equivalent for other languages/
  frameworks in play).

=== Invoked by

* xref:skills/issue.adoc[] — Step 6, when the user chooses automatic implementation over manual review,
  invokes `code` via a `general-purpose` sub-agent so the whole implementation run starts from a clean
  context.
* Also invoked directly by a person via `/code`, once a `plan`-produced `implementation_plan.md` exists at
  the repository root — no orchestrating skill required.

=== Related agents

* `general-purpose` (see xref:agents/overview.adoc[]) — spawned for each project-wide quality/security baseline
  capture and comparison (Steps 3 and 7) and the final build verification. Unlike the previous version of this
  skill, per-task delegation (license headers, Javadoc, scoped tests, coverage, per-task quality) is no longer
  spawned directly by `code` — it happens one level down, inside whichever `<key>-code-one-task` skill handles
  each task.
