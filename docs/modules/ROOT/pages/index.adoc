= AI Catalog

NOTE: This documentation was generated with the assistance of AI. Please report any inaccuracies.

AI Catalog is a collection of https://docs.claude.com/en/docs/claude-code/skills[Claude Code Skills] —
reusable, version-controlled playbooks for common software engineering workflows. Instead of re-explaining
the same process to Claude in every project (how this team bootstraps a Java library, how a pull request
should be reviewed, how release notes get written), each workflow is captured once as a `SKILL.md` file and
invoked on demand with a slash command such as `/plan` or `/pr-review`.

== Why this exists

Prompting an AI assistant to repeat a multi-step process consistently is hard: the steps drift between
sessions, edge cases get forgotten, and every teammate ends up with a slightly different mental model of
"how we do X". A skill fixes that by writing the procedure down once — what to check first, what to skip,
what "done" looks like — so any repository that adopts it gets the same behavior every time it's invoked.

[[core-concepts]]
== Core concepts

=== A skill

Each skill lives in its own directory under `.claude/skills/` and contains a single `SKILL.md` file with:

* A short **description** used by Claude Code to decide when the skill is relevant.
* Step-by-step instructions written for an agent to follow, including what to survey before acting, what
  to leave untouched, and how to report back.

Skills in this catalog range from small, single-purpose checks (`check-license`, `java-test`) to
orchestrators that call other skills in sequence.

=== An agent

This repository does not currently define any custom agents under a `.claude/agents/` directory. Several
skills nevertheless spawn Claude Code's **built-in** agent types through the `Agent` tool, to run a
sub-task in an isolated context (so its detailed output — a full test log, a static-analysis report —
doesn't clutter the main conversation) or to search the codebase without derailing the current flow:

* The **`Explore`** agent — a fast, read-only search agent — is used explicitly by the `explore` skill.
* Every other sub-agent spawned from this catalog (by `code`, `issue`, and the tasks a `plan`-generated
  `implementation_plan.md` describes) uses the **`general-purpose`** agent, Claude Code's default when no
  specific agent type is named.

See xref:agents/overview.adoc[] for detail on where and why each is used.

=== A CLAUDE.md file

`CLAUDE.md` is a plain markdown file that Claude Code loads automatically, without being invoked, at the
start of every session — unlike a skill or agent, which only act when called on. It holds the persistent
facts Claude would otherwise need re-explaining every session: build/test commands, coding conventions,
architectural notes, and "always/never do X" rules specific to a project or a person.

Claude Code reads it from several locations at once (project root, a personal `~/.claude/CLAUDE.md`, nested
subdirectory files, and an enterprise-managed policy file, if any) and concatenates them rather than picking
one — see xref:guides/claude-md.adoc[] for exact locations, precedence, and how to create or update one
without letting it bloat the context window.

=== A plugin

A **plugin** is not a new kind of artifact alongside skills, agents, and CLAUDE.md — it's a packaging and
distribution mechanism for them. A plugin is a directory (with a `.claude-plugin/plugin.json` manifest) that
can bundle any combination of skills, custom agents, slash commands, hooks, and MCP server configs into one
installable, versioned unit, published to a marketplace (a private, organization-internal one or a public
one) and installed into a repository with `/plugin install` instead of copying files by hand.

This repository does not currently package itself as a plugin — its skills are adopted by copying the
`.claude/skills/<name>/` directories directly — but xref:guides/plugins.adoc[] and
xref:guides/packaging-and-distribution.adoc[] cover when a plugin is worth the extra packaging step instead.

== Purpose groups

[[group-bootstrap]]
=== Repository Bootstrap

Scaffolding a brand-new repository's build, docs, CI/CD, and top-level files from nothing, so a new Java
library starts out matching the same conventions as every other repository in the catalog.

[TIP]
====
Reach for this group when standing up a **brand-new repository** — a one-time pass, not something you'd run
again once the repository already exists. `setup-java-library-repository` is the single entry point that
runs the rest in order; each one also works standalone to fill a single gap in a repository that already has
some of this in place (e.g. just `/antora-setup` to add docs to a repository that already has a `pom.xml`).
====

* xref:skills/setup-java-library-repository.adoc[`setup-java-library-repository`] — orchestrator: bootstraps
  a brand-new Java library repository end to end in one pass.
* xref:skills/setup-java-library.adoc[`setup-java-library`] — generates `pom.xml` and the standard Maven
  source layout.
* xref:skills/antora-setup.adoc[`antora-setup`] — scaffolds the Antora documentation site.
* xref:skills/setup-java-gitignore.adoc[`setup-java-gitignore`] — creates or refreshes the root `.gitignore`.
* xref:skills/setup-java-github-workflows.adoc[`setup-java-github-workflows`] — scaffolds/updates the CI/CD

* xref:skills/setup-changelog.adoc[`setup-changelog`] — one-time bootstrap of `CHANGELOG.md`.
* xref:skills/setup-readme.adoc[`setup-readme`] — creates or refreshes the root `README.md`.

[[group-quality]]
=== Java Code Quality

Point-in-time checks against a Maven/Java project, each reporting real, measured results — a coverage
percentage, a pass/fail, a list of lint findings — rather than an estimate.

[TIP]
====
Reach for one of these directly, any time, while **iterating on code day to day** — before committing,
before opening a PR, or just to check where a class currently stands. You rarely need to invoke them
yourself during the xref:index.adoc#group-cycle[Issue-to-PR Development Cycle], though: `code` already fans
out to all four automatically for every task it implements.
====

* xref:skills/java-test.adoc[`java-test`] — runs the JUnit suite via Surefire, scoped to a
  class/method/package/tag selector.
* xref:skills/java-coverage.adoc[`java-coverage`] — runs JaCoCo and reports the exact measured coverage
  percentage for target class(es).
* xref:skills/java-code-quality.adoc[`java-code-quality`] — runs Checkstyle/PMD/SpotBugs and summarizes every
  issue found, grouped by tool.
* xref:skills/java-javadoc.adoc[`java-javadoc`] — audits and completes Javadoc coverage for given classes,
  then verifies it builds cleanly.

[[group-cycle]]
=== Issue-to-PR Development Cycle

Taking a piece of work from "described" — a tracked GitHub issue, or a task you describe yourself — to an
implemented, open pull request.

[TIP]
====
This is the group to reach for on a **day-to-day basis**: picking up an issue or an ad hoc task and carrying
it through branch creation, research, planning, and implementation without re-explaining that process each
time. `issue` is the single entry point for the full pipeline; `explore` and `plan` are also commonly run
individually (`/explore`, `/plan`) ahead of, or instead of, the full `/issue` flow.
====

* xref:skills/issue.adoc[`issue`] — orchestrator: branch, explore, plan, implement, open PR, review — for a
  tracked GitHub issue.
* xref:skills/explore.adoc[`explore`] — read-only research pass over the codebase, optionally grounded in a
  GitHub issue.
* xref:skills/plan.adoc[`plan`] — writes a concrete, reviewable `implementation_plan.md`, grounded in a prior
  `explore` pass.
* xref:skills/code.adoc[`code`] — executes `implementation_plan.md` end to end: implement, test, document,
  and quality-check each task.

[[group-pr]]
=== Pull Request Lifecycle

Everything after code exists but before it's merged: describing the change and getting it reviewed.

[NOTE]
====
Used automatically as the last leg of `issue`'s pipeline once `code` finishes, but just as useful standalone
— e.g. after making changes outside the `/issue` flow and wanting a PR description drafted, or a second,
convention-aware review pass before asking a teammate for one.
====

* xref:skills/pr-description.adoc[`pr-description`] — drafts a PR description from the real diff, and
  optionally opens/updates the pull request.
* xref:skills/pr-review.adoc[`pr-review`] — reviews an open pull request against the linked issue's intent
  and this codebase's conventions.

[[group-maintenance]]
=== Repository Maintenance

Keeping an already-bootstrapped repository correct and current, well after its initial setup.

[NOTE]
====
Intended for whoever **maintains the repository over time** rather than someone mid-feature: cutting
releases, keeping docs from drifting after unrelated code changes, and periodically verifying license
headers.
====

* xref:skills/release.adoc[`release`] — cuts a release branch, bumps versions, drafts release
  notes/changelog, and opens the release PR.
* xref:skills/update-docs.adoc[`update-docs`] — brings the Antora documentation module up to date with
  changes already made to the code.
* xref:skills/check-license.adoc[`check-license`] — verifies (and backfills) license headers against the
  repository's actual license.

== Common workflows

These sections walk through how the catalog's skills compose for a few recurring jobs. Each is a real,
supported path through the graph above — not every skill in the catalog appears in one, since some
(the xref:index.adoc#group-quality[Java Code Quality] skills, `check-license`) are more often invoked as
building blocks of a larger flow than run standalone by a person.

=== Setting up a new repository

`setup-java-library-repository` collects the project's identity and CI/CD parameters once, then
orchestrates six other skills in a fixed order so nothing is asked twice:

[mermaid]
....
flowchart TD
    A["/setup-java-library-repository"] --> B["setup-java-library\n(pom.xml + source layout)"]
    B --> C["antora-setup\n(docs site)"]
    C --> D["setup-java-gitignore\n(.gitignore)"]
    D --> E["setup-java-github-workflows\n(CI/CD)"]
    E --> F["setup-changelog\n(CHANGELOG.md)"]
    F --> G["setup-readme\n(README.md, last — sees everything above)"]
....

Any of these six can also be run standalone against an existing repository to fill a single gap (e.g. just
`/antora-setup` to add docs to a repository that already has a `pom.xml`).

=== Typical development cycle, starting from an issue

`issue` is the single entry point for taking a tracked GitHub issue to an open, reviewed pull request:

[mermaid]
....
flowchart LR
    A["/issue &lt;id&gt;"] --> B["explore\n(understand the issue)"]
    B --> C["plan\n(write implementation_plan.md)"]
    C --> D{Hand off to /code?}
    D -- yes --> E["code\n(implement + test, via a sub-agent)"]
    D -- no --> H[Stop for manual review]
    E --> F["pr-description\n(open PR)"]
    F --> G["pr-review\n(review the new PR)"]
....

`code` itself fans out to the xref:index.adoc#group-quality[Java Code Quality] skills and `check-license`
for every task it implements (each via its own sub-agent, so their detailed output stays out of the main
conversation) — see xref:skills/code.adoc[].`explore` and `plan` can also be run individually
(`/explore`, `/plan`) ahead of, or instead of, the full `/issue` flow.

=== Generating a release

`release` must be run from `develop` and drives the whole version bump itself, delegating only the README
and PR description:

[mermaid]
....
flowchart LR
    A["/release"] --> B["Pick release + next SNAPSHOT version"]
    B --> C["Bump pom.xml, docs/antora.yml, Antora version snippets"]
    C --> D["setup-readme\n(refresh README.md)"]
    D --> E["Write whats-new.adoc + CHANGELOG.md"]
    E --> F["Commit, push release_x.y.z"]
    F --> G["pr-description\n(fill in the release PR)"]
....

=== Updating an existing repository with new functionality

There is no single orchestrator for this — it's a set of independent skills, each scoped to one artifact,
run after code has already changed:

[mermaid]
....
flowchart TD
    Code["Code change already made\n(uncommitted or on a branch)"] --> UD["update-docs\n(Antora pages)"]
    Code --> RM["setup-readme\n(README.md, if it already exists: proposes a diff)"]
    Code --> CLI["check-license\n(license headers on touched files)"]
    UD -.->|only if no Antora module exists yet| AS["antora-setup"]
....

`setup-changelog` is deliberately **not** part of this flow: it only bootstraps a changelog that doesn't
exist yet and stops immediately if `CHANGELOG.md` is already present (see xref:skills/setup-changelog.adoc[]).
Keeping an existing changelog current on each release is `release`'s job, not a standalone update skill's.

== Keeping this documentation current

[IMPORTANT]
====
This documentation site can be created or updated in two ways:

* **Preferred — run the `analyze-claude-skills-and-agents` skill** (`/analyze-claude-skills-and-agents`).
It (re)generates this page — the reference table, purpose groups, dependency graph, and common
workflows above — plus one detail page per skill/agent under `skills/` and `agents/`, directly from
whatever `.claude/skills/*/SKILL.md` files and agent definitions actually exist in the repository.
It's idempotent, so it's safe to re-run any time a skill or agent is added, removed, or changed; see
xref:index.adoc#_keeping_this_documentation_current[Keeping this documentation current] above.
* **Manual edits.** This page and the detail pages are plain AsciiDoc files under
`docs/modules/ROOT/pages/`, and can be edited directly like any other Antora content. This is fine for
wording or structural fixes, but anything describing what a skill or agent actually does should instead
go through the skill above — hand-written content there will be overwritten the next time it runs.
====

This page and every detail page under xref:index.adoc#group-tooling[Catalog Tooling] are themselves
generated — by `analyze-claude-skills-and-agents`, from whatever skills/agents actually exist under
`.claude/skills/` (and `.claude/agents/`, if any) at the time it runs:

[mermaid]
....
flowchart LR
    A["/analyze-claude-skills-and-agents"] --> B["Discover every SKILL.md\n+ agent usage on disk"]
    B --> C["Build the dependency graph\n+ purpose groups + workflows"]
    C --> D["Write index.adoc, skills/*.adoc,\nagents/*.adoc, nav.adoc"]
    D --> E["npx antora — verify the\nsite still builds"]
....

Re-run it (`/analyze-claude-skills-and-agents`) after adding, removing, or materially changing a skill or
agent, so this page and the detail pages it links to don't drift from what's actually on disk.

Each skill's own detail page (linked from the table above) covers its exact inputs, outputs, and execution
flow. Each skill is also self-contained under its own `.claude/skills/<name>/SKILL.md` — copy the ones you
need into another repository's `.claude/skills/` directory to adopt them there.
